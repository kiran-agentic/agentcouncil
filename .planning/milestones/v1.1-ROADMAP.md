### Phase 1: Schemas
**Goal**: Typed data models exist that every downstream component can import and depend on
**Depends on**: Nothing (first phase)
**Requirements**: CONS-01, CONS-02, CONS-04
**Success Criteria** (what must be TRUE):
  1. A Pydantic model for the consensus artifact exists with all required fields: `recommended_direction`, `agreement_points`, `disagreement_points`, `rejected_alternatives`, `open_risks`, `next_action`, `status`
  2. The `status` field is a typed enum with four values: `consensus`, `consensus_with_reservations`, `unresolved_disagreement` (negotiated outcomes), and `partial_failure` (system-assigned when an adapter errors — added in Phase 4)
  3. The consensus artifact serializes to valid JSON that a downstream agent or tool can parse without additional transformation
  4. Importing the schema module raises no errors and type-checks cleanly

### Phase 2: Adapter Layer
**Goal**: Protocol logic can call any LLM backend through a single interface, and the protocol can be tested without invoking real agent CLIs
**Depends on**: Phase 1
**Requirements**: ADAPT-01, ADAPT-02, ADAPT-03, ADAPT-04, ADAPT-05
**Success Criteria** (what must be TRUE):
  1. An adapter interface (abstract class or protocol) defines the contract for making an LLM call and returning a response
  2. A Codex adapter implements the interface for the outside agent role and produces a real response by invoking the local `codex` CLI in non-interactive mode
  3. A Claude adapter implements the interface for the lead agent role by invoking the local `claude` CLI subprocess — not the calling Claude instance — enabling the MCP server to obtain the lead-agent proposal during a single tool invocation
  4. A stub adapter implements the interface and returns deterministic canned responses, making it possible to run the full protocol without any real `claude` or `codex` subprocess
  5. Local v1 usage works with existing Claude Code / ChatGPT CLI login state and does not require manually provisioning API keys
  6. Swapping any adapter requires changing only the constructor — no changes to protocol logic

### Phase 3: Brief Preparation
**Goal**: Claude can produce a structured brief that gives the outside agent full context while provably excluding Claude's own proposed solution
**Depends on**: Phase 2
**Requirements**: BRIEF-01, BRIEF-02, BRIEF-03, BRIEF-04
**Success Criteria** (what must be TRUE):
  1. Given a conversation or repo context, the brief builder produces a structured brief containing problem statement, background, constraints, goals, and open questions
  2. The brief contains no Claude proposal, draft design, or implied direction — this is enforced by an automated contamination check that runs before the brief is sent to the outside agent
  3. When contamination is detected, the brief is rejected with a specific error rather than sent — the outside agent never receives a contaminated brief
  4. When the problem involves code, the brief includes relevant file excerpts or repo context so the outside agent has the information it needs
**Plans**: 1 plan
Plans:

### Phase 4: Deliberation Engine
**Goal**: The full 4-round brainstorm protocol runs to completion, produces a typed consensus artifact reflecting the actual outcome (including failures), and retains the full round transcript
**Depends on**: Phase 3
**Requirements**: PROTO-01, PROTO-02, PROTO-03, PROTO-04, PROTO-05, PROTO-06, CONS-03, CONS-05
**Success Criteria** (what must be TRUE):
  1. Calling `brainstorm(brief)` with the stub adapter runs all four rounds in the correct order — briefing → outside proposal → lead proposal → negotiation — and returns a consensus artifact
  2. Each round receives only the context it is permitted to see: the outside agent sees only the brief, not Claude's proposal; Claude sees the outside agent's response before forming its own proposal
  3. When negotiation fails to produce synthesis, the artifact has `status: unresolved_disagreement` and is returned as a valid result — the function does not retry or force a false consensus
  4. When the outside agent errors, the returned artifact reflects the partial failure state with an appropriate status rather than returning only Claude's perspective
  5. The full round transcript — brief sent, outside agent proposal, lead proposal, negotiation output — is returned as part of the response artifact, so the caller has provenance without a separate query mechanism

- [x] 04-02-PLAN.md — TDD: brainstorm() function implementing the 4-round protocol with context isolation and failure handling

### Phase 5: MCP Server + Skill-Driven Path
**Goal**: Claude Code can call `brainstorm` as an MCP tool in a single invocation, receiving a complete consensus artifact from local CLI-backed agents, with no stdout corruption of the JSON-RPC stream. Additionally, a skill-driven path enables Claude Code to act as orchestrator AND lead agent directly, calling Codex via `outside.py`.
**Depends on**: Phase 4
**Requirements**: MCP-01, MCP-02, MCP-03, MCP-04, REAL-01, REAL-02, REAL-03, REAL-04, REAL-05
**Dual architecture**:
  - **MCP path** (`server.py`): Spawns both agent subprocesses (Haiku for brief extraction, Opus for lead, Codex for outside). For programmatic/non-Claude-Code clients.
  - **Skill path** (`.claude/skills/brainstorm/SKILL.md` + `outside.py`): Claude Code is orchestrator AND lead. Calls Codex via `python3 -m agentcouncil.outside`. ~17s faster per round.
**Success Criteria** (what must be TRUE):
  1. An MCP server exposes a `brainstorm` tool that Claude Code can discover and call
  2. A single call to the `brainstorm` tool runs the full deliberation protocol and returns the consensus artifact — the caller manages no multi-turn state
  3. The MCP server uses stdio transport and can be launched as a local subprocess from Claude Code without additional infrastructure
  4. All logging — including debug output, errors, and trace messages — routes exclusively to stderr; nothing writes to stdout except the JSON-RPC responses, verified by running the server and inspecting its stdout
  5. Real end-to-end validation: at least one real brainstorm run through the actual Claude Code → MCP → AgentCouncil → `codex` CLI / `claude` CLI path produces a consensus artifact with `consensus` status
  6. At least one real run produces `consensus_with_reservations` status
  7. At least one real run produces `unresolved_disagreement` status or a partial-failure artifact
  8. At least one real run is invoked from the actual Claude Code MCP integration, not from an isolated local test harness
  9. v1 cannot ship with only stubbed integration tests — real-run validation is a ship criterion, not a stretch goal
**Plans**: 2 plans

## Phase Details

### Phase 6: Common Deliberation Framework
**Goal**: All deliberation functions share a common envelope, transcript format, session lifecycle, and error handling — so Review, Decide, and Challenge deliver consistent structure without reimplementing protocol mechanics
**Depends on**: Phase 5 (builds on existing adapter layer, session management patterns, and MCP server)
**Requirements**: COM-01, COM-02, COM-03, COM-04, COM-05, COM-06, COM-07, COM-08, COM-09, COM-10, COM-11, COM-12, COM-13
**Success Criteria** (what must be TRUE):
  1. Pydantic schemas exist for the common envelope (`DeliberationResult` with `deliberation_status` + function-specific `artifact` + `transcript`) and `TranscriptTurn` with `role`, `content`, optional `source_refs`, and the transcript carries `input_prompt`, `outside_initial`, `lead_initial`, `exchanges`, `final_output` fields
  2. The dual-independent protocol runs end-to-end with stub adapters: both agents receive the same factual input, neither sees the other's initial analysis, and the result contains a valid `deliberation_status` from the `ConsensusStatus` enum
  3. One persistent Codex MCP session is created per invocation, used for all outside-agent turns, and closed on both success and failure paths — never reused across invocations
  4. Adapter/session failures after protocol start produce a `partial_failure` result with transcript populated through the last successful stage; input validation failures abort before any agent call; JSON parse or schema-validation failure on final synthesis returns `unresolved_disagreement` with raw output preserved
  5. All output models survive JSON round-trip serialization with no field loss, and outcome/disposition invariants are enforced in Pydantic validators (not only prompt instructions)

- [x] 06-02-PLAN.md — TDD: CodexSession context manager + dual-independent protocol runner with error handling

### Phase 7: Review
**Goal**: Users can submit an artifact and receive a structured, evaluative critique with severity-rated findings from independent reviewers who confirm or dispute each other's observations
**Depends on**: Phase 6
**Requirements**: REV-01, REV-02, REV-03, REV-04, REV-05, REV-06, REV-07, REV-08, REV-09, REV-10, REV-11
**Success Criteria** (what must be TRUE):
  1. Calling `review(artifact)` with stub adapters returns a `ReviewArtifact` containing a `verdict` (pass/revise/escalate), `summary`, `findings[]` with severity and confidence, `strengths[]`, `open_questions[]`, and `next_action`
  2. Each finding carries `id`, `title`, `severity` (critical/high/medium/low), `impact`, `description`, `evidence`, `locations[]`, `confidence` (high/medium/low), `agreement` (confirmed/disputed), `origin` (outside/lead/both), and `source_refs[]` — with optional `priority` (P1/P2/P3) only when `artifact_type=code`
  3. Review output is evaluative only: findings state impact but contain no fixes or implementation recommendations; the lead can confirm, dispute, or add omitted findings; disputed findings appear in the final output (not collapsed into consensus)
  4. The director's brief includes changed files, requirements implemented, test files, and plan docs as factual context — no opinion or assessment leaks into what the outside agent sees

- [x] 07-02-PLAN.md — TDD: review() function, evaluative-only prompts, MCP review tool, /review skill (REV-07..11)

### Phase 8: Decide
**Goal**: Users can submit a decision question with options and receive a structured comparison with per-option assessments, a constrained outcome (decided/deferred/experiment), and clear traceability of assumptions, tradeoffs, and confidence
**Depends on**: Phase 6
**Requirements**: DEC-01, DEC-02, DEC-03, DEC-04, DEC-05, DEC-06, DEC-07, DEC-08, DEC-09, DEC-10, DEC-11, DEC-12
**Success Criteria** (what must be TRUE):
  1. Calling `decide(decision, options)` with stub adapters returns a `DecideArtifact` containing `outcome` (decided/deferred/experiment), `winner_option_id`, `decision_summary`, `option_assessments[]`, `defer_reason`, `experiment_plan`, `revisit_triggers[]`, and `next_action`
  2. Outcome invariants hold in all cases: if decided, exactly one option has `disposition=selected` and `winner_option_id` matches; if deferred, zero options selected and `defer_reason` non-empty; if experiment, zero selected, at least one viable, and `experiment_plan` non-empty
  3. The outside agent evaluates only caller-provided options (cannot invent new ones), and the output includes assumptions, tradeoffs, and confidence even when a single winner is selected
  4. The director's brief includes project constraints, relevant code/docs, and criteria — the lead's preference or ranking does not appear in what the outside agent sees

### Phase 9: Challenge
**Goal**: Users can submit a plan or design and receive an adversarial stress test that identifies failure modes, tests assumptions, and delivers a readiness verdict — with independent attack and defense phases
**Depends on**: Phase 6
**Requirements**: CHL-01, CHL-02, CHL-03, CHL-04, CHL-05, CHL-06, CHL-07, CHL-08, CHL-09, CHL-10, CHL-11, CHL-12, CHL-13
**Success Criteria** (what must be TRUE):
  1. Calling `challenge(artifact)` with stub adapters returns a `ChallengeArtifact` containing `readiness` (ready/needs_hardening/not_ready), `summary`, `failure_modes[]`, `surviving_assumptions[]`, `break_conditions[]`, `residual_risks[]`, and `next_action`
  2. Readiness invariants hold: if ready, no failure mode has `disposition=must_harden` (including `failure_modes=[]` for no-credible-attack); if needs_hardening or not_ready, at least one failure mode has `disposition=must_harden`
  3. The outside agent's first turn is an independent attack and the lead's first turn is an independent defense — both produced before seeing each other's output; exchange rounds attack the lead's defense arguments, not a modified artifact
  4. Challenge is adversarial only: it attacks assumptions and does not propose repairs; the director's brief includes the target plan, assumptions, success criteria, and relevant code — not the lead's confidence level or defense strategy
  5. Default 2 rounds of exchange (vs 1 for Review and Decide), reflecting the adversarial nature of the function
**Plans**: 2 plans
