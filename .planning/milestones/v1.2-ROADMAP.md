### 🚧 v1.2 Multi-Backend & OSS Release (In Progress)

**Milestone Goal:** Expand AgentCouncil to arbitrary outside agent backends with a read-only deliberation runtime for raw LLMs, then ship the first public release.

## Phase Details

### Phase 10: Config System
**Goal**: Users can configure named backend profiles and AgentCouncil resolves the correct backend at runtime
**Depends on**: Phase 9
**Requirements**: CFG-01, CFG-02, CFG-03, CFG-04, CFG-05, CFG-06, TEST-02
**Success Criteria** (what must be TRUE):
  1. User can define a named backend profile in `.agentcouncil.json` with provider, model, endpoint, and api_key_env fields and AgentCouncil loads it without error
  2. Running `show-effective-config` reports each resolved value alongside its source level (skill arg / project config / global config / env var / default)
  3. Config precedence applies correctly: skill arg overrides project config, project config overrides global, env var overrides default
  4. Legacy `AGENTCOUNCIL_OUTSIDE_AGENT` env var still selects a backend when no profile config exists
  5. Config files containing raw API key strings are rejected — only env var names are accepted
**Plans**: 2 plans

Plans:
- [x] 10-01-PLAN.md — Core config module: BackendProfile, AgentCouncilConfig, ProfileLoader with TDD tests
- [x] 10-02-PLAN.md — show-effective-config MCP tool with source attribution

### Phase 11: Provider ABC + Read-Only Tool Harness
**Goal**: The OutsideProvider contract is defined and the read-only deliberation harness is secure and resilient
**Depends on**: Phase 10
**Requirements**: PROV-01, PROV-06, PROV-08, RUN-01, RUN-02, RUN-03, RUN-04, RUN-05, RUN-06, RUN-07, TEST-03, TEST-04
**Success Criteria** (what must be TRUE):
  1. A StubProvider implementing the OutsideProvider ABC can run a full tool loop through OutsideRuntime without error
  2. Any file path returned by an LLM tool call that escapes the workspace (path traversal, symlink escape) is rejected before execution
  3. Requests to read `.env`, `.pem`, or `*.key` files are blocked by the extension allowlist
  4. A model that exceeds the retry limit (3 tool-call attempts per turn) falls back to text-only mode gracefully
  5. Both native function-calling format and the textual action protocol (`READ_FILE path=...`) are handled by the runtime
**Plans**: 3 plans

Plans:
- [x] 11-01-PLAN.md — OutsideProvider ABC, response models, StubProvider, pyproject extras
- [x] 11-02-PLAN.md — OutsideRuntime with security, tool loop, budget, and retry
- [x] 11-03-PLAN.md — Textual action protocol and integration degradation tests

### Phase 12: OutsideRuntime + OutsideSession
**Goal**: The full provider-runtime-session stack composes correctly, backward compatibility is preserved, and transcript metadata is populated
**Depends on**: Phase 11
**Requirements**: SESS-01, SESS-02, SESS-03, SESS-04, META-01, META-02, META-03, TEST-05, TEST-06
**Success Criteria** (what must be TRUE):
  1. OutsideSession opens, accepts call(), and closes with correct lifecycle semantics using StubProvider
  2. OutsideSessionAdapter wraps OutsideSession behind the existing AgentAdapter interface so existing callers require no changes
  3. All four protocol engines populate `outside_provider`, `outside_profile`, `outside_model`, `outside_session_mode`, and `outside_workspace_access` fields in returned transcripts
  4. Importing AgentAdapter directly triggers a DeprecationWarning at import time
  5. All 273+ existing tests continue passing without modification
**Plans**: 2 plans

Plans:
- [x] 12-01-PLAN.md — OutsideSession lifecycle, OutsideSessionAdapter shim, AgentAdapter deprecation warning
- [x] 12-02-PLAN.md — TranscriptMeta extension and metadata propagation through all protocol engines

### Phase 13: Ollama + OpenRouter Providers
**Goal**: Raw LLMs reachable via Ollama and OpenRouter can participate in deliberations
**Depends on**: Phase 12
**Requirements**: PROV-02, PROV-03
**Success Criteria** (what must be TRUE):
  1. An Ollama model listed locally can complete a single brainstorm round via OllamaProvider with tool calling enabled and `stream=False`
  2. An OpenRouter model can complete a single brainstorm round via OpenRouterProvider using the `openai` SDK with the OpenRouter base_url and a Bearer token from env var
  3. Both providers call `auth_check()` before the first deliberation and surface actionable error messages when credentials or connectivity are missing
**Plans**: 2 plans

Plans:
- [x] 13-01-PLAN.md — OllamaProvider implementation, tests, pyproject.toml ollama extra
- [x] 13-02-PLAN.md — OpenRouterProvider implementation, tests, __init__.py re-exports

### Phase 14: Bedrock Provider
**Goal**: AWS Bedrock models can participate in deliberations with correct credential handling
**Depends on**: Phase 13
**Requirements**: PROV-04, PROV-05, PROV-07, TEST-01
**Success Criteria** (what must be TRUE):
  1. A Bedrock-hosted model completes a single brainstorm round via BedrockProvider using the `converse()` API wrapped in `asyncio.to_thread()`
  2. The `toolUse`/`toolResult` Bedrock naming is normalized to the internal format so the runtime tool loop functions without special-casing
  3. A new boto3 client is created per invocation, so STS credential expiry mid-session does not cause a failure
  4. Contract tests for all three providers (Ollama, OpenRouter, Bedrock) validate request/response format against stubs or mocks
**Plans**: 1 plan

Plans:
- [x] 14-01-PLAN.md — BedrockProvider implementation, message normalization, contract tests

### Phase 15: MCP Integration + Skill Migration
**Goal**: All deliberation tools and skills use the uniform session API with no backend-specific branching
**Depends on**: Phase 14
**Requirements**: MCP-01, MCP-02, MCP-03, MCP-04, MCP-05, SKIL-01, SKIL-02, SKIL-03
**Success Criteria** (what must be TRUE):
  1. Calling `outside_start(prompt, profile="ollama-llama3")` returns a session_id and first model response
  2. `outside_reply(session_id, prompt)` continues the session and `outside_close(session_id)` cleans up resources
  3. `get_outside_backend_info(profile="ollama-llama3")` returns capability flags without opening a session
  4. Each of the 4 skills (brainstorm/review/decide/challenge) accepts an optional `backend=` argument that selects a named profile
  5. Skills use `outside_start/reply/close` exclusively — no provider-specific branching remains in skill code
**Plans**: 2 plans

Plans:
- [x] 15-01-PLAN.md — Provider factory, session registry, and 4 new MCP session tools
- [x] 15-02-PLAN.md — Skill migration to uniform session API with async brainstorm
**UI hint**: yes

### Phase 16: Conformance Certification
**Goal**: Any backend can be certified for protocol compatibility and gates are enforced before deliberations run
**Depends on**: Phase 15
**Requirements**: CERT-01, CERT-02, CERT-03, CERT-04
**Success Criteria** (what must be TRUE):
  1. Running ConformanceCertifier against a StubProvider produces a certification result with tool use, multi-turn coherence, structured output, and budget adherence scores
  2. Certification results are cached by `(model_id, provider_version, agentcouncil_version)` so repeated runs skip re-testing
  3. A stale certification triggers a printed warning that includes the re-certification command
  4. Attempting to run review or challenge with a prompt-only model is blocked with an actionable error message
**Plans**: 2 plans

Plans:
- [x] 16-01-PLAN.md — Certifier module: CertificationResult, CertificationCache, ConformanceCertifier, stale warning
- [x] 16-02-PLAN.md — Protocol gates in review_tool and challenge_tool with integration tests
