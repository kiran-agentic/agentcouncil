# Requirements: AgentCouncil

**Defined:** 2026-04-10
**Core Value:** Independence before convergence — agents must think independently before seeing each other's proposals, producing genuinely better decisions

## v1 Requirements

Requirements for initial release. The first wedge: `brainstorm` as a first-class AgentCouncil function.

### Brief Preparation

- [x] **BRIEF-01**: Claude can extract a structured brief from conversation or repo context containing problem statement, background, constraints, goals, and open questions
- [x] **BRIEF-02**: Brief must explicitly exclude Claude's preferred solution, draft design, or implied direction
- [x] **BRIEF-03**: Automated contamination detection validates that no proposed solution leaks into the brief before it reaches the outside agent
- [x] **BRIEF-04**: Brief includes relevant file excerpts or repo context when the problem requires code awareness

### Deliberation Protocol

- [x] **PROTO-01**: Brainstorm follows a deterministic 4-round protocol: briefing → outside proposal → lead proposal → negotiation
- [x] **PROTO-02**: Outside agent (Codex) produces its first proposal from the brief alone, without seeing any Claude proposal or direction
- [x] **PROTO-03**: Claude develops its own proposal only after seeing the outside agent's independent response
- [x] **PROTO-04**: Negotiation round: outside agent compares both proposals, identifies agreements, disagreements, and proposes synthesis
- [x] **PROTO-05**: Protocol ordering is enforced in code, not by trust — each round receives only the context it is permitted to see
- [x] **PROTO-06**: Unresolved disagreement is a valid terminal outcome — the system must not force synthesis when negotiation fails to converge

### Consensus Artifact

- [x] **CONS-01**: Brainstorm returns a structured typed artifact with fields: `recommended_direction`, `agreement_points`, `disagreement_points`, `rejected_alternatives`, `open_risks`, `next_action`, `status`
- [x] **CONS-02**: Status field supports four states: `consensus`, `consensus_with_reservations`, `unresolved_disagreement` (negotiated outcomes), and `partial_failure` (system-assigned only when an adapter errors — never a valid negotiation result)
- [x] **CONS-03**: If the outside agent errors or negotiation fails, the artifact reflects the partial failure state rather than silently returning only Claude's perspective
- [x] **CONS-04**: Artifact is machine-parseable (typed JSON/Pydantic) for downstream agent consumption
- [x] **CONS-05**: Round transcript returned as part of the response artifact — what brief was sent, what the outside agent proposed, what the lead proposed, what negotiation produced — so the caller has provenance without a separate query mechanism

### MCP Integration

- [x] **MCP-01**: AgentCouncil exposes a `brainstorm` tool via an MCP server callable from Claude Code
- [x] **MCP-02**: Entire brainstorm protocol completes within a single MCP tool invocation (caller does not manage multi-turn state)
- [x] **MCP-03**: MCP server uses stdio transport for local Claude Code integration
- [x] **MCP-04**: All logging routes to stderr — no stdout writes that would corrupt the JSON-RPC stream

### Agent Adapter

- [x] **ADAPT-01**: Agent adapter interface abstracts LLM backend calls so protocol logic is not coupled to a specific provider
- [x] **ADAPT-02**: Codex adapter implements the adapter interface for the outside agent role using the local `codex` CLI in non-interactive mode
- [x] **ADAPT-03**: Claude adapter implements the adapter interface for the lead agent role using the local `claude` CLI subprocess — used in the MCP path only. In the skill-driven path, Claude Code itself acts as lead agent directly (no subprocess), which is faster and gives the lead full conversation context
- [x] **ADAPT-04**: Stub adapter exists for testing the protocol without invoking real agent CLIs
- [x] **ADAPT-05**: Local v1 operation reuses existing CLI authentication/session state for `claude` and `codex` and does not require manually provisioning API keys

### Real-Run Validation

- [x] **REAL-01**: v1 completion requires successful real end-to-end brainstorm runs through the actual Claude Code → MCP → AgentCouncil → Codex path, not only stubbed integration tests
- [x] **REAL-02**: At least one real run ends in `consensus` status
- [x] **REAL-03**: At least one real run ends in `consensus_with_reservations` status
- [x] **REAL-04**: At least one real run ends in `unresolved_disagreement` or produces a partial-failure artifact
- [x] **REAL-05**: At least one real run is invoked from the actual Claude Code MCP integration, not only from an isolated local test harness

## v1.1 Requirements

### Common Deliberation Framework

- [x] **COM-01**: System exposes `review`, `decide`, and `challenge` as distinct functions with separate skills and Pydantic output schemas
- [x] **COM-02**: All functions return a common envelope: `deliberation_status` + function-specific `artifact` + shared `transcript`
- [x] **COM-03**: `deliberation_status` uses ConsensusStatus values: consensus, consensus_with_reservations, unresolved_disagreement, partial_failure (system-only)
- [x] **COM-04**: Each artifact carries a function-specific outcome field: Review has `verdict`, Decide has `outcome`, Challenge has `readiness`
- [x] **COM-05**: Transcript uses shared `TranscriptTurn` with `role`, `content`, optional `source_refs`; transcript fields are `input_prompt`, `outside_initial`, `lead_initial`, `exchanges`, `final_output`
- [x] **COM-06**: During the independent phase, both agents see the same factual input but neither sees the other's initial analysis (dual-independent, not outside-first)
- [x] **COM-07**: One persistent Codex MCP session per invocation, created before first outside turn, closed on success or failure, never reused across invocations
- [x] **COM-08**: Adapter/session failures after protocol start return `partial_failure` with transcript populated through last successful stage — never raise to caller
- [x] **COM-09**: Final JSON parse or schema-validation failure returns `unresolved_disagreement` with raw output preserved in `transcript.final_output`
- [x] **COM-10**: `partial_failure` excluded from any schema shown to models during final synthesis
- [x] **COM-11**: Input validation failures abort before any agent call
- [x] **COM-12**: All outputs survive JSON round-trip serialization with no field loss
- [x] **COM-13**: Outcome/disposition invariants enforced in Pydantic validation, not only in prompt instructions

### Review

- [x] **REV-01**: `ReviewInput` accepts `artifact` (text or file content) + optional `artifact_type` (code/design/plan/document/other, default other) + optional `review_objective`, `focus_areas`, `rounds`
- [x] **REV-02**: `ReviewArtifact` includes `verdict`, `summary`, `findings[]`, `strengths[]`, `open_questions[]`, `next_action`
- [x] **REV-03**: `verdict` is one of: pass, revise, escalate
- [x] **REV-04**: Each finding includes `id`, `title`, `severity`, `impact`, `description`, `evidence`, `locations[]`, `confidence`, `agreement`, `origin`, `source_refs[]`
- [x] **REV-05**: `severity` is critical/high/medium/low; optional `priority` P1/P2/P3 only when `artifact_type=code`
- [x] **REV-06**: `confidence` is high/medium/low; `agreement` is confirmed/disputed; `origin` is outside/lead/both
- [x] **REV-07**: Review is evaluative only — findings state impact, not fixes or implementation recommendations
- [x] **REV-08**: Lead can confirm, dispute, or add omitted findings before finalization
- [x] **REV-09**: Disputed findings preserved in output, not collapsed into consensus
- [x] **REV-10**: Director's brief includes changed files, requirements implemented, test files, plan docs — factual context only, no opinion or assessment
- [x] **REV-11**: Default 1 round

### Decide

- [x] **DEC-01**: `DecideInput` requires `decision` question + `options[]` with at least 2 entries each having `id`, `label`, `description` + optional `criteria`, `constraints`, `rounds`
- [x] **DEC-02**: `DecideArtifact` includes `outcome`, `winner_option_id`, `decision_summary`, `option_assessments[]`, `defer_reason`, `experiment_plan`, `revisit_triggers[]`, `next_action`
- [x] **DEC-03**: `outcome` is one of: decided, deferred, experiment
- [x] **DEC-04**: Each option assessment includes `option_id`, `pros`, `cons`, `blocking_risks`, `disposition`, `confidence`, `source_refs[]`
- [x] **DEC-05**: `disposition` is selected/viable/rejected/insufficient_information
- [x] **DEC-06**: If outcome=decided, exactly one option has disposition=selected and winner_option_id matches
- [x] **DEC-07**: If outcome=deferred, zero options selected, defer_reason non-empty
- [x] **DEC-08**: If outcome=experiment, zero options selected, at least one viable, experiment_plan non-empty
- [x] **DEC-09**: Outside agent evaluates only caller-provided options — cannot invent new options
- [x] **DEC-10**: Must include assumptions, tradeoffs, and confidence even in single-winner mode
- [x] **DEC-11**: Director's brief includes project constraints, relevant code/docs, criteria — not lead's preference or ranking
- [x] **DEC-12**: Default 1 round

### Challenge

- [x] **CHL-01**: `ChallengeInput` requires `artifact` (target plan/design/approach) + optional `assumptions[]`, `success_criteria`, `constraints`, `rounds`
- [x] **CHL-02**: `ChallengeArtifact` includes `readiness`, `summary`, `failure_modes[]`, `surviving_assumptions[]`, `break_conditions[]`, `residual_risks[]`, `next_action`
- [x] **CHL-03**: `readiness` is one of: ready, needs_hardening, not_ready
- [x] **CHL-04**: Each failure mode includes `id`, `assumption_ref`, `description`, `severity`, `impact`, `confidence`, `disposition`, `mitigation`, `source_refs[]`
- [x] **CHL-05**: `severity` is critical/high/medium/low; `disposition` is must_harden/monitor/mitigated/accepted_risk/invalidated
- [x] **CHL-06**: If readiness=ready, no failure mode has disposition=must_harden
- [x] **CHL-07**: If readiness=needs_hardening or not_ready, at least one failure mode has disposition=must_harden
- [x] **CHL-08**: Outside agent's first turn is independent attack; lead's first turn is independent defense — both produced before seeing each other's output
- [x] **CHL-09**: Exchange rounds attack the lead's defense arguments, not a modified artifact
- [x] **CHL-10**: `no_credible_attack` represented as failure_modes=[] and readiness=ready
- [x] **CHL-11**: Challenge is adversarial only — attacks assumptions, does not propose repairs (can request stress tests in followups)
- [x] **CHL-12**: Director's brief includes target plan, assumptions, success criteria, relevant implementation code — not lead's confidence level or defense strategy
- [x] **CHL-13**: Default 2 rounds

## v2 Requirements

The delivery system foundation. These extend AgentCouncil from a deliberation tool into the autonomous system described in VISION.md.

### Vision File

- **VIS-01**: First-class vision artifact format with defined schema (product goal, target user, constraints, scope boundaries, quality bar, non-goals, success criteria, known risks, implementation context)
- **VIS-02**: Vision file validation — system rejects malformed or incomplete vision artifacts with specific error messages
- **VIS-03**: Vision file versioning — changes to the vision are tracked with timestamps, diffs, and the reason for change
- **VIS-04**: Vision file is the authoritative source for all downstream planning; requirements, phases, and verification trace back to it

### Human Approval and Escalation

- **ESCL-01**: System defines explicit escalation triggers: ambiguous product decisions, irreversible actions, budget/time threshold breaches, unresolved disagreements that block progress
- **ESCL-02**: Escalation pauses autonomous work and presents the human with structured context: what happened, what the options are, what the system recommends, and why
- **ESCL-03**: Human can approve, reject, redirect, or defer at any escalation gate
- **ESCL-04**: Escalation decisions are recorded in project memory with rationale

### Persistent Project Memory

- **MEM-01**: Durable storage of all council decisions: what was decided, what alternatives were rejected, what risks were known, who proposed what
- **MEM-02**: Memory persists across sessions — a new session can reconstruct the full decision history
- **MEM-03**: Current plan state (active phase, completed tasks, blocked items, open risks) is always queryable
- **MEM-04**: Memory supports both structured queries ("what was decided about auth?") and timeline queries ("what happened in phase 2?")

### Task Graph and Decomposition

- **TASK-01**: System can decompose a vision or decision into a task graph with milestones, subtasks, and dependencies
- **TASK-02**: Each task has: objective, inputs, constraints, expected outputs, verification criteria, estimated complexity
- **TASK-03**: Dependencies between tasks are explicit and enforced — a task cannot start if its dependencies are incomplete
- **TASK-04**: Task graph identifies checkpoints where councils should reconvene before proceeding

### Delegation Contract

- **DELG-01**: Every delegated task includes a delegation contract: objective, inputs, constraints, expected outputs, verification criteria, return artifact format
- **DELG-02**: Delegated agent operates within the boundaries of its contract — scope creep beyond the contract triggers escalation
- **DELG-03**: Return artifacts are validated against the contract's expected output schema before acceptance
- **DELG-04**: Delegation tracks task ownership — which agent was assigned, when, and what they returned

### Execution and Verification Loop

- **EXEC-01**: Agents can perform real work against the codebase: implementation, documentation, test writing, debugging, configuration
- **EXEC-02**: Verification checks work against the original vision and phase goals, not only tests and lint
- **EXEC-03**: Failed verification triggers reconvene — the council reassesses rather than the executor retrying blindly
- **EXEC-04**: Verification produces a structured report: what was checked, what passed, what failed, what needs human judgment

### Reconvene Triggers

- **RECV-01**: Council reconvenes after task completion to assess whether the phase goal was met
- **RECV-02**: Council reconvenes after failed verification to decide on corrective action
- **RECV-03**: Council reconvenes after unresolved disagreement that blocks downstream work
- **RECV-04**: Council reconvenes when plan drift is detected — execution results diverge from the plan beyond a defined threshold
- **RECV-05**: Reconvene produces a structured decision: continue, revise plan, escalate to human, or abandon and re-plan

### Agent Role System

- **ROLE-01**: Named reusable roles: architect, challenger, implementer, reviewer, verifier (extensible)
- **ROLE-02**: Roles define the agent's perspective, constraints, and evaluation criteria for a given function
- **ROLE-03**: Team configurations compose roles for specific patterns: architect+challenger, planner+executor+verifier, researcher+decision-maker
- **ROLE-04**: Roles persist across sessions — the same team can be reconvened with the same perspective

### Budget, Time, and Autonomy Limits

- **LIMIT-01**: Max deliberation rounds per brainstorm/council (prevents infinite negotiation)
- **LIMIT-02**: Max token spend per session and per phase (prevents cost explosion)
- **LIMIT-03**: Max task depth (prevents infinite decomposition into sub-sub-sub-tasks)
- **LIMIT-04**: Max unattended duration — system must escalate to human if no human interaction for a configurable period
- **LIMIT-05**: All limits are configurable with sensible defaults; breaching a limit triggers escalation, not silent failure

### Auditability and Provenance

- **AUDIT-01**: Every significant decision records: who proposed it, what alternatives were considered, what was rejected and why, what risks were known at decision time
- **AUDIT-02**: Decision provenance is traceable from the final artifact back through negotiation, proposals, and the original brief
- **AUDIT-03**: Audit trail is durable and queryable — a human can reconstruct the reasoning chain for any decision after the fact
- **AUDIT-04**: Provenance includes which agent played which role and what context each agent had access to

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| YAML workflow runner / pipeline engine | Not the product identity — AgentCouncil sells deliberation protocol and delivery outcomes, not pipeline definitions |
| Claude-proposes-first pattern | Validated by manual testing to cause anchoring bias — violates core principle |
| Generic AgentOS / ungrounded agent swarm | Research shows up to 17.2x error amplification in unstructured agent networks |
| Majority voting / model voting wrapper | Voting skips the synthesis step — AgentCouncil requires actual negotiation |
| Copying AG2/OpenHands/GSD product model | Reference for subsystems only, not product identity |
| Silent failure on partial errors | Deceptive — artifact must reflect actual state including failures |
| Web UI or standalone application (v1) | Claude Code is the host environment; MCP is the integration surface |
| Multi-vendor federation (v1) | A2A protocol support deferred until brainstorm and delivery loop are proven |
